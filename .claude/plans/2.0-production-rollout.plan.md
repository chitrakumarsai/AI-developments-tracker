# Plan — Phase 2 Production Rollout (planned in 2.0)

> Ordered, gated plan for shipping AI Chronicles to production as a multi-user,
> authenticated product. Pairs with `.claude/prds/2.0-production.prd.md`.
> **Two gates per slice** (§12.3): GATE 1 = approve plan before code; GATE 2 =
> approve diff before commit. Every auth/user-data/ingestion change gets a
> `security-reviewer` pass.

**Locked (2.0 checkpoint):** magic-link + GitHub + Google auth · shared catalog +
private per-user personalization (RLS) · separate dedicated prod Supabase project ·
2.0 = planning docs only, provision in 2.3.

---

## Target data model (the RLS contract)

Global (owner-curated, read-by-all): **`sources`**, **`items`**, **`source_candidates`**.
Per-user (private, `user_id`-scoped): **`feedback`**, **`saved_views`**,
**`app_settings`**, and read-state.

| Table | Ownership | RLS read | RLS write |
|---|---|---|---|
| `sources` | global | all authenticated | owner/service-role only |
| `items` | global | all authenticated | service-role only (ingestion) |
| `source_candidates` | global | owner only | owner only |
| `feedback` | per-user | `user_id = auth.uid()` | `user_id = auth.uid()` |
| `saved_views` | per-user | `user_id = auth.uid()` | `user_id = auth.uid()` |
| `app_settings` | per-user (was singleton) | `user_id = auth.uid()` | `user_id = auth.uid()` |
| read-state | per-user | `user_id = auth.uid()` | `user_id = auth.uid()` |

**Schema migration implied:** add `user_id uuid references auth.users` to
`feedback`, `saved_views`, `app_settings`, read-state; convert `app_settings` from
a singleton row to one-row-per-user; backfill the owner's existing rows to the
owner `user_id`. Ranking that consumes feedback must scope to `auth.uid()`.

## Slice order (each = its own preference-checkpoint → PRD → plan → TDD → review → PR)

### 2.1 — Authentication (standard+; security-reviewer required)
**Goal:** users can sign in; the app knows who they are; unauth users are handled.
1. Enable Supabase Auth providers: email magic-link, GitHub OAuth, Google OAuth
   (dev project first). Register OAuth apps, store client IDs/secrets in env.
2. Auth UI: a sign-in page with the three methods; sign-out; session display.
   Follow ECC `react/security` (httpOnly cookies via Supabase SSR helpers — never
   localStorage sessions).
3. Route protection: middleware gating the app behind a session; public sign-in
   route. Server components read the session server-side.
4. Owner bootstrap: designate the first/owner account (seed or first-sign-in rule).
5. Tests: session presence/absence, redirect behavior, provider callback happy-path.
**Outcome:** individual users can sign in. No per-user data yet (still shared).

### 2.2 — Per-user data + RLS (standard+; database-reviewer + security-reviewer)
**Goal:** enforce the RLS contract above; feeds personalize per account.
1. Migration: add `user_id` to the four per-user tables; convert `app_settings`
   to per-user; backfill owner rows. Versioned in `supabase/migrations/`.
2. Enable RLS + write explicit policies on every user-scoped table (table above).
3. Update typed queries to write `auth.uid()` and rely on RLS for reads (drop the
   relaxed single-user assumptions from Phase 1).
4. Scope ranking/feedback + saved views + settings to the current user.
5. Tests: **cross-user isolation** (user A cannot read/write user B's rows),
   owner-only source writes, ingestion still writes items via service-role.
**Outcome:** every user gets a feed tuned to them; no cross-user leakage.

### 2.3 — Deployment & hardening (large; security-reviewer + performance-optimizer)
**Goal:** live, secure, fast — on a dedicated prod project. **First provisioning.**
1. Create the **dedicated prod Supabase project**; promote all migrations dev→prod;
   configure prod Auth providers + redirect URLs.
2. Vercel prod: set env vars (anon key client-side; service-role + OpenAI + source
   tokens + `CRON_SECRET` server-only), wire prod branch → production deploy.
3. **Email hardening:** replace Supabase's test mailer with a real SMTP provider
   (Resend/Postmark/SES) for magic-link deliverability.
4. **Abuse controls:** CAPTCHA (Turnstile/hCaptcha) on sign-up + Supabase auth
   rate limits.
5. Security headers + production CSP (ECC web `security`); verify service-role key
   never reaches the client; startup validation of required secrets.
6. Cron in prod (Vercel Cron → ingestion route, `CRON_SECRET`-guarded).
7. Perf pass: mobile-network budget, CWV targets (ECC web `performance`).
8. Full `security-reviewer` + `database-reviewer` sweep before go-live.
**Outcome:** production website — live, secure, fast.

### 2.4 — Discoverability & growth (standard; per feature)
**Goal:** others can find/onboard; sources self-suggest; in-app source management.
1. SEO + sharing + new-user onboarding (ECC `seo`).
2. Automated source discovery (citation/mention-trail; periodic "new candidates"
   digest) — builds on Phase 1's `source_candidates` + rating gate.
3. In-app **Source Management UI** (add/pause/re-weight/archive without code).
**Outcome:** the site grows and sources are managed in-app.

## Auth scaling path (baked in from the 2.0 discussion)

Keep the three chosen methods; treat "scaling auth" as **2.3 hardening**, not more
login buttons: production SMTP → CAPTCHA → rate limits. Add **Apple** only if/when
an iOS app ships (App Store rule); **LinkedIn/X** if the audience wants them;
**TOTP MFA** once accounts hold value; **SAML SSO** only when a real org asks (B2B).

## Cross-cutting guardrails

- **Secrets (§21):** anon key = only client key; service-role server-only; validate
  at startup; `.env` gitignored; prod secrets in Vercel/Supabase settings.
- **Migrations:** single source of truth in `supabase/migrations/`; promote, never
  hand-edit prod.
- **Untrusted content (§12.7):** ingestion sanitization unchanged; auth callbacks +
  form actions validated (Zod) and rate-limited.
- **Reviews:** `security-reviewer` mandatory on 2.1/2.2/2.3; `database-reviewer` on
  every RLS/migration change.

## Sequencing & dependencies

```
2.1 auth ──► 2.2 per-user + RLS ──► 2.3 deploy + harden ──► 2.4 growth
(sign-in)    (isolation)            (go-live, provision)     (SEO, discovery, UI)
```
2.2 depends on 2.1 (needs `auth.uid()`). 2.3 depends on 2.2 (won't expose un-isolated
data publicly). 2.4 is post-launch.
