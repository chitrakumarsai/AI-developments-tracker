# Plan — Phase 2.2: Per-user data + RLS

> Ordered, gated build to partition personalization per account and enforce it with
> Row Level Security. Pairs with `.claude/prds/2.2-per-user-rls.prd.md`. **Two
> gates** (§12.3). `database-reviewer` **and** `security-reviewer` are mandatory
> before GATE 2. Branches off `feat/2.1-authentication` (needs `auth.uid()`).

**Locked (2.2 checkpoint):** backfill Phase-1 data to the owner · end-to-end RLS
with an anon-read policy on `sources`/`items` (reads leave service-role) ·
`profiles` table with a `role` column, owner set from `OWNER_EMAIL`.

---

## Target RLS contract (final)

| Table | Read | Write |
|---|---|---|
| `sources` | anon + authenticated | owner only (`profiles.role='owner'`) |
| `items` | anon + authenticated | service-role only (ingestion) |
| `source_candidates` | owner only | owner only |
| `profiles` | own row (`id = auth.uid()`) | own row; role not self-elevatable |
| `feedback` | `user_id = auth.uid()` | `user_id = auth.uid()` |
| `saved_views` | `user_id = auth.uid()` | `user_id = auth.uid()` |
| `app_settings` | `user_id = auth.uid()` | `user_id = auth.uid()` |
| read-state | `user_id = auth.uid()` | `user_id = auth.uid()` |

RLS is default-deny: every allowed verb needs an explicit policy. Ingestion keeps
the service-role client (bypasses RLS) for `items` writes only.

## Build order

### Step 1 — `profiles` table + upsert on sign-in
1. Migration: `profiles(id uuid PK references auth.users on delete cascade, email text, role text not null default 'member', created_at)`. RLS: a user reads/updates only their own row; **`role` is not self-elevatable** (update policy excludes `role`, or role is set only server-side).
2. App: after a session is established (session/callback path), upsert the user's
   profile; set `role='owner'` when the email is in `OWNER_EMAIL`, else `'member'`.
   Reuse `roleForEmail` from 2.1 — single source of truth for owner derivation.
3. Tests: owner email → `role='owner'`; other → `'member'`; a user cannot elevate
   their own role.

### Step 2 — Add `user_id` + convert `app_settings`; backfill owner
1. Migration: add `user_id uuid references auth.users` to `feedback`, `saved_views`,
   `app_settings`, and read-state; index each.
2. Convert `app_settings` from singleton to one-row-per-user (drop the singleton
   constraint; key by `user_id`).
3. **Backfill (idempotent):** assign `user_id IS NULL` rows in `feedback`,
   `saved_views`, and the `app_settings` singleton to the resolved owner id
   (looked up via `profiles` where `role='owner'` / `OWNER_EMAIL`).
4. Tests: backfill assigns exactly the null-owner rows to the owner; re-running is a
   no-op.

### Step 3 — Enable RLS + policies (the isolation core)
1. Enable RLS on `profiles`, `feedback`, `saved_views`, `app_settings`, read-state,
   `sources`, `items`, `source_candidates`.
2. Write explicit policies per the contract table: per-user `USING`/`WITH CHECK`
   on `user_id = auth.uid()`; anon+auth read on `sources`/`items`; owner-only writes
   on `sources`/`source_candidates` via `profiles.role='owner'`; service-role (which
   bypasses RLS) remains ingestion's `items` writer.
3. `database-reviewer` pass on the migration + policies.

### Step 4 — Switch reads to the RLS client; scope writes to the user
1. Feed/read queries (`getFeedItems`, settings, saved views, feedback reads) move
   from the service-role client to the **auth-aware SSR/anon client**, relying on
   RLS. Keep ingestion + cron on service-role.
2. Writes (feedback, saved views, settings) set/rely on `auth.uid()` and drop the
   Phase-1 single-user assumptions.
3. Ranking that consumes feedback scopes to the current user.
4. Tests + typecheck + build green.

### Step 5 — Isolation test suite (the headline proof)
1. **Cross-user:** user A cannot read or write user B's feedback / saved views /
   settings / read-state (each verb).
2. **Owner-only:** a non-owner cannot write `sources`/`source_candidates`; the owner
   can.
3. **Ingestion:** service-role still writes `items`; cron path unaffected.
4. **Anon:** anonymous read of `sources`/`items` succeeds via the anon client; anon
   cannot read any per-user table.
5. `security-reviewer` pass focused on isolation + service-role confinement.

## Gates & reviews

- **GATE 1:** approve this PRD + plan (now).
- **`database-reviewer`** (mandatory): migration safety, policy completeness
  (every verb), indexes on `user_id`, `USING`/`WITH CHECK` correctness, backfill.
- **`security-reviewer`** (mandatory): cross-user isolation, no service-role in the
  read path, role not self-elevatable, anon scope limited to global tables.
- **GATE 2:** approve the diff before commit; conventional commits; PR links to this
  PRD + plan. Per the standing workflow, push after commit and boot the UI to review.

## Cross-cutting guardrails (§21, §12.7)

- Single migrations source of truth in `supabase/migrations/`; never hand-edit prod.
- Service-role stays server-only and confined to ingestion; anon key is the only
  client key; RLS is the enforcement boundary, not UI gating.
- Validate inputs at write boundaries (Zod) as today.

## Definition of done

`user_id` on all per-user tables; `app_settings` per-user; `profiles` + owner role;
RLS enabled with complete policies; feed reads via the RLS client (no service-role
in reads); owner's Phase-1 data backfilled; **cross-user isolation proven by tests**;
`database-reviewer` + `security-reviewer` pass; typecheck + build + tests green.

## Sequencing

```
Step1 profiles ─► Step2 user_id+backfill ─► Step3 RLS policies ─► Step4 switch reads ─► Step5 isolation tests
```
