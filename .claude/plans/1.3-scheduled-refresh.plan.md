## Design

Extract the ingest loop (today inline in the manual route) into one shared, injectable
helper so the cron route, the manual route, and the tests all share one code path — and
so the resilience + due-filter live in exactly one place (DRY, §coding-style).

```
                         ┌─ GET  /api/cron/refresh ─┐   (Vercel Cron, daily)
 isAuthorizedCron(req) ──┤                          ├──► runIngestion(client, { dueOnly:true })
                         └─ POST /api/ingest/run ───┘   (manual, source=<id|all>)
                                                              │  per-source try/catch
                                                              │  dueOnly → isDue(interval)
                                                              ▼
                                                        persistItems (already stamps last_fetched)
                                                              ▼
                                                        RunSummary { added, sources, perSource[] }
                                                              ▼
                                                    console.info(JSON) + JSON response
```

Tolerates `"1 day"`, `"2 days"`, `"HH:MM:SS"`, `"1 day 02:00:00"`, null. Unparseable/never-fetched → **due** (fail-safe: refresh rather than silently skip). |

## Risks / notes

- **Interval parsing:** PostgREST returns `interval` as Postgres text (`"1 day"`, `"24:00:00"`), not ISO-8601. Parser targets exactly the formats we seed; anything else → fail-safe "due". Covered by tests.
- **No migration** → zero Supabase-CLI friction (per project memory, cloud sync is REST-only). Due-math lives in JS, fully unit-tested.
- Cron won't actually fire until the app is deployed (Phase 2.3); this slice makes it correct-on-deploy and locally testable via the authenticated GET.
