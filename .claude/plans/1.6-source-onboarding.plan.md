# 1.6 — Source onboarding: candidate → rate → promote (slice 1)

**Subphase:** 1.6 (Source discovery & onboarding) — first thin vertical.
**Preference-checkpoint:** user chose "Manual candidate → rate → promote" (defer paste-list auto-extraction), 2026-07-04.
**Tier:** large — ingests untrusted external content (feed validation fetches a user-supplied URL) → **mandatory `security-reviewer` / SSRF care** (§12.5, §12.7). TDD throughout.

## Goal
Exercise the whole §5 rating gate end-to-end for ONE source: add a candidate → it lands in
`source_candidates` (state=suggested) → rate it (skip/keep/⭐) → promote a rated candidate to
`sources` (status=active) only after its feed/URL validates. Auto-extraction from a pasted
list is a follow-up slice.

## Reuse (already in repo)
- `lib/ingestion/net.ts` → `unsafeUrlReason(url)` (SSRF guard: http/https only, blocks
  private/loopback/link-local/IMDS), `FETCH_TIMEOUT_MS`, `USER_AGENT`.
- `lib/ingestion/rss/rss.ts` → `parseRssFeed(xml, ref)` to confirm a feed actually parses.
- `lib/ingestion/sanitize.ts` → `sanitizeText` / `sanitizeUrl` for untrusted candidate fields.
- Patterns to mirror: `lib/views/persist.ts` (server-only + injectable client) and
  `app/api/views/route.ts` (thin Zod route + `{success,data,error}` envelope).

## Data model
No migration — `source_candidates (id, platform, handle_or_url, why_suggested, sample_items
jsonb, rating int, state suggested|promoted|rejected, created_at)` and `sources` both exist.

## Logic libs (server-only, injectable client, TDD)
- `lib/candidates/persist.ts`
  - `createCandidate({platform, handleOrUrl, whySuggested?})` → sanitize + insert (state=suggested).
  - `listSuggested()` → candidates in the rating queue (state=suggested), newest first.
  - `reviewCandidate(id, {action:'rate'|'skip', rating?})` → rate sets `rating`; skip sets
    state=rejected. Throws on DB error.
  - `promoteCandidate(id, {name, category, ingestionType, tags})` → inserts a `sources` row
    (status=active) from the candidate + sets candidate state=promoted. (Validation happens in
    the route BEFORE this, so this stays a pure DB op.)
- `lib/candidates/validate.ts`
  - `validateFeedUrl(url): Promise<{ok:boolean; reason?:string; sampleCount?:number}>` —
    `unsafeUrlReason` → `fetch` (timeout + UA) → `parseRssFeed` → ok when ≥1 item parses.
    Never throws; returns a reason string on any failure. (api/scrape types: validate URL
    safety + reachability only, skip RSS parse — RSS is the default and the manual path.)

## Routes (thin, Zod, envelope)
- `POST /api/candidates` — `{platform, handleOrUrl, whySuggested?}` → createCandidate.
- `POST /api/candidates/review` — `{id:uuid, action:'rate'|'skip', rating?:1..5}` → reviewCandidate.
- `POST /api/candidates/promote` — `{id:uuid, name, category, ingestionType, tags?}` →
  `validateFeedUrl` first; on failure return 422 `{reason}`; on success `promoteCandidate`.

## UI — new `/sources` route
- `app/sources/page.tsx` (server): `listSuggested()`; renders the add form + rating queue.
  Resilient (DB error → notice). Linked from the main header nav.
- `components/sources/AddCandidateForm.tsx` (client): platform + URL/handle + why → POST →
  `router.refresh()`; visible error.
- `components/sources/CandidateCard.tsx` (client): shows candidate; skip / keep / ⭐ buttons
  (POST review); a Promote control (name + category + type + tags) that POSTs promote and
  surfaces the validation reason on 422. `router.refresh()` on success.

## Acceptance
- Add a candidate → appears in the `/sources` rating queue.
- Skip → it leaves the queue (rejected); rate/⭐ → rating stored.
- Promote a rated candidate with a REACHABLE feed → new `sources` row (active), candidate=promoted,
  and its items start appearing after the next ingest.
- Promote with an unreachable/unsafe/non-feed URL → blocked with a clear reason (422), no source created.
- SSRF: private/loopback/non-http URLs rejected by `unsafeUrlReason` before any fetch.
- tsc + eslint clean; unit tests green; `security-reviewer` pass (no CRITICAL/HIGH).

## Known follow-ups (security — GATE 2, 2026-07-04)
- **Redirect-based SSRF (MEDIUM):** `validateFeedUrl`'s `fetch` follows redirects by
  default, so a public URL could 30x→private/loopback host, bypassing the pre-fetch
  `unsafeUrlReason` guard. Accepted for Phase-1 (local, single-user); before Phase 2 /
  any public deploy, set `redirect: "manual"` and re-run the SSRF guard on each hop (or
  reject 3xx). Sits alongside the CSRF / rate-limit / user-scoping Phase-2 deferrals.
