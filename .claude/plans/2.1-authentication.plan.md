## Delta to the 2.0 RLS contract

The "public feed" choice amends one row of the 2.0 data-model table:

| Table | 2.0 contract | 2.1 amendment |
|---|---|---|
| `sources` | authenticated read | **anon + authenticated read** |
| `items` | authenticated read | **anon + authenticated read** |
| everything else | unchanged | unchanged |

Per-user tables (`feedback`, `saved_views`, `app_settings`, read-state) stay
**auth-only, `user_id`-scoped** ÔÇö RLS enforcement still lands in **2.2**, not here.
In 2.1 we only open anon *read* on the two global tables; we do **not** yet add
`user_id` columns or per-user policies.

## Build order (TDD: red ÔåÆ green ÔåÆ refactor per step)

### Step 1 ÔÇö Supabase Auth config (dev project)
- Enable providers in the dev Supabase project: email magic-link, GitHub OAuth,
  Google OAuth. Register the GitHub + Google OAuth apps; store client IDs/secrets in
  `.env` (server-only) and document in `.env.example` (no values).
- Add `OWNER_EMAIL` to `.env` / `.env.example`.
- Configure allowed redirect URLs (local dev origin + `/auth/callback`).
- *No app code yet ÔÇö config + env only.*

### Step 2 ÔÇö Supabase SSR client + session plumbing
- Add server/browser clients via `@supabase/ssr` in `web/lib/supabase/` (cookie-based).
- `middleware.ts`: refresh the session cookie on each request (`updateSession`); it
  does **not** gate routes (public feed stays public).
- A server helper `getSessionUser()` returning `{ user, role }` where
  `role = email === OWNER_EMAIL ? 'owner' : 'member'` (derived server-side).
- **Tests:** helper returns null for anon; returns `owner` for `OWNER_EMAIL`;
  returns `member` otherwise. (Pure role-derivation unit-tested; session read mocked.)

### Step 3 ÔÇö `/sign-in` page (editorial, mobile-first)
- Route `web/app/sign-in/page.tsx` matching the AI Chronicles design language
  (tokens, Inter/Fraunces, touch-friendly targets). Three actions: Continue with
  Google, Continue with GitHub, and an email field ÔåÆ Send magic link.
- Server actions (or route handlers) initiate each flow; `redirectTo` validated
  against a same-origin path allowlist (guards open-redirect).
- **Tests:** renders all three methods; magic-link action validates email; invalid
  `redirectTo` is rejected/normalized.

### Step 4 ÔÇö OAuth/magic-link callback + sign-out
- `web/app/auth/callback/route.ts`: exchange code for session, set httpOnly cookie,
  redirect to validated return-to (default `/`).
- Sign-out action clears the session; a small identity affordance in the header
  (signed-out ÔåÆ "Sign in" link; signed-in ÔåÆ email/avatar + "Sign out").
- **Tests:** callback happy-path sets session and redirects; bad/missing code fails
  soft to `/sign-in`; sign-out clears session.

### Step 5 ÔÇö Anon-aware personalization affordances
- Personalization entry points (feedback thumbs, save-view, settings) check the
  session: signed-in ÔåÆ act as today; anon ÔåÆ link to `/sign-in?redirectTo=<current>`.
- **No data partitioning yet** ÔÇö signed-in writes still hit the shared plane (2.2
  isolates them). This step only stops anon users from hitting write paths and nudges
  them to sign in.
- **Tests:** anon feedback control routes to `/sign-in` with return-to; signed-in
  path unchanged.

### Step 6 ÔÇö Anon read policy on the two global tables
- Since Phase 1 RLS is relaxed/disabled, ensure the **public feed keeps working for
  anon** under the SSR anon client. If a migration is needed to add an explicit
  anon-read policy on `sources`/`items`, add it in `supabase/migrations/` (narrowly
  scoped to those two tables). `database-reviewer` checks it.
- **Tests / manual:** anon `GET /` returns items; no other table is anon-readable.

## Cross-cutting guardrails (┬º21, ┬º12.7)

- No hardcoded secrets; validate required env (`OWNER_EMAIL`, OAuth creds) at startup.
- Anon key is the only client key; service-role stays server-side.
- Validate all auth inputs (Zod) ÔÇö email, `redirectTo`, callback params.
- Dev-project + test-mailer only in 2.1; SMTP/CAPTCHA/rate-limits are 2.3.

## Definition of done

Anon can browse the shared feed; all three sign-in methods work end-to-end with
httpOnly-cookie sessions; owner resolves via `OWNER_EMAIL`; sign-out works; anon
personalization nudges to `/sign-in`; `security-reviewer` + `database-reviewer`
pass; tests green (80%+ on new logic). **No per-user isolation yet ÔÇö that's 2.2.**
