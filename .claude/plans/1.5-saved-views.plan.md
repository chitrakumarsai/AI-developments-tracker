# 1.5 — Saved filter views (§8.3)

**Subphase:** 1.5, slice 1 of 2 to close the subphase (then a11y+perf+design polish).
**Preference-checkpoint:** user chose "Saved views + polish pass" to complete 1.5, 2026-07-04.
**Tier:** standard — writes user data (`saved_views`), but non-sensitive; inline security check (no auth yet, Phase 1 single-user).

## Decisions (why)
- **No migration** — `saved_views (id, name, filters jsonb, created_at)` already exists in the hosted DB (init migration).
- **URL-as-state stays the source of truth.** A saved view just stores the current filter params as jsonb; **loading a view = navigating to `feedHref(filters)`**. No new client store, no per-item state.
- **Whitelist on read.** Stored `filters` jsonb is data read back into a URL, so `viewToHref` picks only known keys (section/sort/window/source/tag/q/state) and coerces them — never blind-spreads jsonb into a link (untrusted-data discipline, §12.7).
- **One client island.** `SavedViewsBar` (client) receives the server-fetched view list + the current filters, does `fetch` + `router.refresh()` for save/delete so the page stays put. Loading a view is a plain `<Link>` (server-navigable).

## Files
- `lib/views/persist.ts` (server-only, injectable client): `createView({name,filters})`, `listViews()`, `deleteView(id)` — throw on DB error.
- `lib/views/href.ts` (client-safe): `viewToHref(filters: unknown): string` + `type SavedFilters` — whitelist/coerce known keys → `feedHref`. Pure, testable.
- `app/api/views/route.ts`: `POST` Zod `{ name: string(1..80), filters: SavedFilters }` → createView. Envelope.
- `app/api/views/delete/route.ts`: `POST` Zod `{ id: uuid }` → deleteView. Envelope.
- `components/feed/SavedViewsBar.tsx` (client): renders saved-view `<Link>`s (via `viewToHref`) each with a delete ✕; a "Save this view" inline form (name input) shown when any filter is active. Optimistic-ish: `router.refresh()` after write; visible error on failure.
- `app/page.tsx`: server-fetch `listViews()`, build `currentFilters` from parsed params, render `<SavedViewsBar>` above the feed.

## Read/query path
- No change to `getFeedItems` — views only produce URLs that the existing param pipeline already handles.

## Acceptance
- With filters active, "Save this view" persists a named preset; it appears in the bar and survives reload.
- Clicking a saved view applies exactly those filters (URL changes, feed updates).
- Deleting a view removes it; page stays put.
- Junk/unknown keys in a stored view can't produce a malformed or unsafe URL.
- tsc + eslint clean; all tests green.

## Out of scope (later)
- Per-user views (Phase 2 auth + RLS + `saved_views.user_id`).
- Editing/renaming a view (delete + re-save for now).
- Reordering / default view.
