# PRD — Phase 2.0: Production Rollout Plan

> Problem-first PRD for taking AI Chronicles from a local single-user tool to a
> deployed, authenticated, multi-user product. **Phase 2.0 delivers the plan, not
> the infrastructure** — provisioning happens in 2.3. This doc pairs with
> `.claude/plans/2.0-production-rollout.plan.md`.

**Status:** Draft — awaiting GATE 1 approval
**Subphase:** 2.0 (Production planning)
**Author:** Claude + owner preference-checkpoint (2026-07-05)

---

## 1. Problem

Phase 1 shipped a working AI radar that runs locally for one user with auth
disabled and Supabase RLS relaxed. It cannot be shared: there are no accounts, no
per-user data isolation, no production database, and no deploy pipeline for real
traffic. To let anyone other than the owner use it, we need authentication,
per-user data separation enforced at the database, a dedicated production
environment, and the operational hardening (email, abuse controls, monitoring)
that real sign-ups require.

## 2. Why now

Phase 1 (1.0–1.6) plus the app-feedback-v3 polish are merged and the product is
proven day-to-day by the owner. The feature surface is stable enough that the
risky, one-way-door decisions of production (auth model, data-isolation model,
prod topology) can be made deliberately before writing any Phase 2 code.

## 3. Goals

- Produce an **approved, ordered rollout plan** for Phase 2 (2.1 → 2.4).
- Lock the **auth model**, the **multi-tenancy / RLS data model**, the **prod
  Supabase topology**, and the **secrets + monitoring approach** — in writing.
- De-risk the one-way doors: decide the shape now; build later.

## 4. Non-goals (explicitly deferred)

- **No prod infrastructure is provisioned in 2.0** (owner-selected scope). The
  prod Supabase project and Vercel prod settings are stood up in 2.3.
- No auth code, RLS migrations, or deploy config written yet — those are 2.1–2.3.
- Automated source discovery and the in-app Source Management UI remain in 2.4.

## 5. Locked decisions (from the 2.0 preference-checkpoint)

| Decision | Choice | Consequence |
|---|---|---|
| **Auth methods** | Magic link (email) + GitHub OAuth + Google OAuth | Supabase Auth providers; no password storage. Apple/SSO deferred. |
| **Catalog model** | **Shared radar + private per-user personalization** | `sources`/`items` stay global; `feedback`, `saved_views`, `app_settings`, read-state become per-user, enforced by RLS. |
| **Prod DB** | **Separate dedicated prod Supabase project** | Two projects (dev + prod); migrations promote dev → prod. Real user data isolated from dev. |
| **2.0 scope** | **Planning docs only; provision in 2.3** | 2.0 output is this PRD + the rollout plan. |

## 6. Users & core value

- **Primary:** the owner (unchanged) — now signing in instead of running headless.
- **New:** additional individuals who want the same curated AI radar. Each gets
  the **same shared feed of sources/items**, but their **likes, saved views,
  settings, and read-state are private to them** and shape *their* ranking.

The product promise is unchanged — "the one thing worth reading right now" — now
personalized per account on top of a shared, owner-curated source catalog.

## 7. Success criteria (for the Phase 2 macro-phase, tracked from here)

- A new user can sign in with any of the three methods and see the shared feed.
- No user can read or write another user's feedback/views/settings (RLS-proven).
- Ingestion cost stays flat as users are added (shared catalog, single pipeline).
- Production is live on Vercel + a dedicated Supabase prod project, secure and
  fast on mobile networks.

## 8. Key risks this plan must address

1. **Data leakage between users** — mitigated by RLS on every user-scoped table,
   verified by the `database-reviewer` + `security-reviewer` and explicit tests.
2. **Email deliverability at scale** — Supabase's built-in mailer is test-grade;
   magic link demands a real SMTP provider before public sign-ups (2.3 hardening).
3. **Abuse / bot sign-ups** — CAPTCHA + auth rate limits before launch.
4. **Migration drift dev → prod** — a single migrations source of truth, promoted,
   never hand-edited in prod.
5. **Secret exposure** — service-role key server-only; anon key is the only
   client key; validated at startup.

## 9. Open questions to resolve during 2.1–2.3 planning (not blockers for 2.0)

- Owner-bootstrap: how the first admin/owner account is designated (seed vs. first
  sign-in) and whether any table needs an `owner`/`role` concept for source curation.
- Whether Phase 2 keeps a single shared `sources` writer (owner-only) — likely yes,
  given the shared-catalog choice — and how that's enforced (RLS write policy).
- Email provider choice (Resend vs. Postmark vs. SES) — decided in 2.3.
