# PRD — Phase 2.1: Authentication

> Problem-first PRD for adding sign-in to AI Chronicles. Pairs with
> `.claude/plans/2.1-authentication.plan.md`. Sits under the Phase 2 rollout
> (`.claude/plans/2.0-production-rollout.plan.md`); this slice delivers auth only —
> **no per-user data isolation yet** (that is 2.2).

**Status:** Draft — awaiting GATE 1 approval
**Subphase:** 2.1 (Authentication)
**Author:** Claude + owner preference-checkpoint (2026-07-05)
**Tier:** standard+ · `security-reviewer` pass required (auth surface)

---

## 1. Problem

Phase 1 shipped a single-user radar with auth disabled and RLS relaxed. Before the
app can serve anyone but the owner, it must know **who** a visitor is. Today there
is no sign-in, no session, no notion of an account or an owner. 2.1 introduces
authentication and identity — the foundation 2.2 (per-user RLS) and 2.3 (deploy)
build on. It deliberately stops short of isolating data: after 2.1 every signed-in
user still shares the same relaxed data plane; 2.1's job is only to establish
*identity* and *session*, safely.

## 2. Locked decisions (from the 2.1 preference-checkpoint, 2026-07-05)

| Decision | Choice | Consequence |
|---|---|---|
| **Access model** | **Public feed; sign-in only for personalization** | Anon visitors browse the shared feed read-only. Sign-in gates likes, saved views, settings, and tuned ranking. **Changes the 2.0 RLS contract:** `sources`/`items` become **anon + authenticated read** (was authenticated-only). |
| **Owner bootstrap** | **`OWNER_EMAIL` env allowlist** | On sign-in, an email matching `OWNER_EMAIL` is granted the `owner` role (source-catalog write); everyone else is `member`. Explicit, reassignable, no dev/prod seed drift. |
| **Sign-in UX** | **Dedicated `/sign-in` page** | Full-page editorial route with all three methods; clean OAuth redirect/callback home; mobile-first layout. |
| **Auth methods** *(from 2.0)* | Magic link + GitHub OAuth + Google OAuth | Supabase Auth providers; no password storage. |
| **Session** *(ECC-locked)* | httpOnly cookies via Supabase SSR helpers | Never localStorage; server components read the session server-side. |

## 3. Goals

- A visitor can **sign in** via magic link, GitHub, or Google.
- The app **knows the current user** server-side (session in httpOnly cookies).
- The **owner** is identified via `OWNER_EMAIL` and carries an `owner` role.
- **Anon browsing still works** — the public shared feed renders without a session.
- Personalization affordances (like / save view / settings) **prompt sign-in** when
  used by an anon visitor, instead of failing.
- Sign-out works; the current identity is visible in the UI.

## 4. Non-goals (deferred)

- **Per-user data isolation / RLS enforcement** — 2.2. In 2.1 feedback/views/settings
  remain the shared single-plane from Phase 1 (still behind sign-in to *use*, but not
  yet partitioned by `user_id`).
- **Production provisioning** (prod Supabase project, real SMTP, CAPTCHA, rate limits)
  — 2.3. 2.1 runs against the existing dev Supabase project with its test mailer.
- Roles beyond `owner`/`member`; TOTP/MFA; SSO — later, per the 2.0 auth scaling path.

## 5. Users & core value

- **Anon visitor** — lands on the shared feed, reads freely, is nudged to sign in
  only when they try to personalize.
- **Signed-in member** — same feed, but their personalization actions are now tied
  to an identity (isolation lands in 2.2).
- **Owner** (`OWNER_EMAIL`) — additionally may curate the source catalog.

## 6. Success criteria

- Anon `GET /` renders the shared feed (no redirect, no crash).
- All three sign-in methods complete and land the user back on the app with a
  session cookie set (httpOnly).
- `OWNER_EMAIL` account resolves to `role = owner`; any other to `role = member`.
- Sign-out clears the session; the app returns to the anon view.
- Personalization controls, when anon, route to `/sign-in` (with return-to) rather
  than erroring.
- `security-reviewer` pass: no secrets client-side (anon key only), session in
  httpOnly cookies, OAuth `redirectTo` validated, callback route hardened.

## 7. Key risks

1. **Anon-read RLS regression** — opening `sources`/`items` to anon must not also
   expose the future per-user tables. The anon policy is scoped to exactly the two
   global tables; personalization tables stay auth-only. Verified in 2.2 isolation
   tests; the anon policy is written narrowly here.
2. **Open-redirect via `redirectTo`** — OAuth/magic-link return URLs must be
   validated against an allowlist of same-origin paths.
3. **Session in the wrong store** — must use Supabase SSR cookie helpers; a stray
   `createBrowserClient` persisting to localStorage is a finding.
4. **Owner spoofing** — `OWNER_EMAIL` is a server-only env var; the role is derived
   server-side from the verified session email, never from client input.
5. **Service-role leak** — auth flows use the anon key on the client; the
   service-role key never reaches the browser.

## 8. Open questions (resolve during 2.1 planning, not blockers)

- Do we surface a lightweight `role` concept in code now (owner vs member) or defer
  the *enforcement* of owner-only catalog writes to 2.2's RLS? (Lean: derive role in
  2.1 for UI/gating; enforce writes via RLS in 2.2.)
- Middleware vs. per-route session reads for the small set of gated actions — decided
  in the plan (lean: middleware refreshes the session cookie; gating is per-action).
