# PRD — Subphase 1.2: More Sources (Slice 1 — Catalog runner + generic RSS + company blogs)

**Status:** Draft (awaiting GATE 1 approval)
**Spec:** [`CLAUDE.md`](../../CLAUDE.md) §4, §6, §7 · **Roadmap:** [`ROADMAP.md`](../../ROADMAP.md) 1.2 · **Plan:** [`1.2-more-sources.plan.md`](../plans/1.2-more-sources.plan.md)

## Problem

1.1 proved the pipeline with a single hardcoded source type (arXiv). The ingest route dispatches every active source to `arxivConnector` by name. To aggregate across categories (CLAUDE.md §3) the pipeline must become **catalog-driven**: what runs is decided by the `sources` table, not by code. And most sources are plain RSS — building a bespoke connector per blog would violate DRY.

## Goal

Make ingestion catalog-driven, and add the first non-paper category — **Company & lab blogs** — with **zero new code per RSS source**. After this slice, a new RSS source is added by inserting one `sources` row; the feed shows a mix of Research Papers + Companies & Labs.

## Scope (this slice)

1. A **connector registry** that dispatches by `ingestion_type` (`rss` → generic RSS connector).
2. A **generic RSS connector** — generalized from the existing arXiv connector (which is already essentially generic). All `rss` sources (arXiv *and* company blogs) run through it.
3. The ingest runner uses the registry instead of hardcoding arXiv.
4. Seed a starter set of **company/lab blog RSS feeds** (category `Companies & Labs`), each validated to actually fetch.
5. Feed renders the mixed categories (the category eyebrow already exists on the card).

## Non-goals (deferred)

- API/scrape connectors — **GitHub, Hugging Face, Hacker News, Reddit, YouTube, newsletters** → later 1.2 slices (keyed ones now unblocked since credentials exist).
- X / LinkedIn direct ingest → via secondary sources, later.
- Functional category **filtering / chips / search** → 1.5.
- Scheduled refresh → 1.3. LLM summaries/scoring → 1.4.

## Success criteria

1. The ingest route dispatches by `ingestion_type` via the registry — no source type named inline.
2. arXiv still ingests exactly as before (no regression) through the generic RSS connector.
3. At least 3–4 company/lab blog feeds ingest real items tagged `Companies & Labs`; each seeded feed is verified to fetch.
4. The feed shows a **mix** of Research Papers + Companies & Labs items.
5. Adding a new RSS source requires **only** a `sources` row — no code change.
6. Generic RSS connector + registry have unit tests; existing 1.1 tests still pass; ≥80% coverage on new/changed `lib/ingestion` code.
7. All fields sanitized (untrusted §12.7); SSRF guard retained; no `dangerouslySetInnerHTML`.

## Risks

- **Blog RSS URLs drift / some labs have no clean feed** (we hit this with arXiv). → validate each feed at build time; only seed ones that work; note the rest as candidates.
- **Full-HTML `content:encoded`** in blog feeds → keep link-first: sanitized snippet only, never store/render raw HTML.
- **Regression risk** folding arXiv into the generic connector → covered by keeping the existing arXiv fixture test green.

## Acceptance demo

Reseed (arXiv + blogs) → `POST /api/ingest/run?source=all` → feed shows both Research Papers and Companies & Labs items, each linking to its source → re-run dedupes to 0 → inserting a new blog `sources` row and re-running ingests it with no code change.
