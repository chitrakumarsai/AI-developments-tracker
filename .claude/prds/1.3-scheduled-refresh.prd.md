# PRD — Subphase 1.3: Scheduled Refresh

> Status: GATE 1 (planning). Slice tier: **standard** + `security-reviewer` (cron is an
> untrusted external trigger surface, §12.5). Supersedes the skipped-over 1.3 in
> `phase-1.plan.md`.

## Problem

Content only updates when someone manually POSTs `/api/ingest/run`. The feed goes
stale. §8.4 requires automatic daily / two-day refresh via Vercel Cron.

## Outcome

An authenticated cron endpoint that Vercel fires **once a day** and that refreshes
each active source **only when it is due** per its own `refresh_interval`. One broken
source must not abort the run. No manual action required.

## Preferences captured (preference-checkpoint, §12.4)

| Decision | Choice |
|---|---|
| **Cadence** | Per-source. Vercel Cron ticks daily (`0 6 * * *` UTC); the route runs a source only when `now() − last_fetched ≥ refresh_interval`. Honors per-source overrides (arXiv=1d runs daily; a 2-day source runs every other tick). |
| **Cron auth** | Accept Vercel's `Authorization: Bearer $CRON_SECRET` **and** the existing `x-cron-secret` header (fallback / manual trigger). Reject everything else with 401. |
| **Observability** | Lightweight. Reuse the `last_fetched` stamp already written by `persistItems`; per-source error captured in-memory; return a JSON run summary and emit one structured `console.info` line. **No new table.** |

## Scope (this slice)

- `GET /api/cron/refresh` — authenticated; runs **due** active sources; returns a run summary.
- `web/vercel.json` — `crons` entry (Vercel root dir = `web/`).
- Shared `runIngestion` helper with **per-source try/catch resilience**; both the cron
  route and the existing manual route use it (resolves the `TODO(1.3)` in the manual route).
- Shared cron-auth guard used by both routes.
- Due-filter based on `refresh_interval` vs `last_fetched` (JS interval parser — **no migration**).
- `CRON_SECRET` documented in `.env.example`.

## Out of scope

- New `ingestion_runs` table / durable run history (chose lightweight).
- New connector types (`scrape`/`manual`).
- Any change to ranking, feed UI, or the schema.
- Actually *deploying* to Vercel — cron only fires once deployed (Phase 2.3). This slice
  wires it so it works the moment the app is deployed; locally it's exercised via the
  authenticated GET.

## Acceptance criteria (from `phase-1.plan.md`)

1. Content updates on schedule with no manual action → `vercel.json` cron + route.
2. A deliberately broken source does not abort the whole run; its failure is logged and
   reported in the summary → per-source try/catch.
3. The cron endpoint rejects unauthenticated calls → shared auth guard returns 401.
4. Due-filter respects per-source `refresh_interval`.
5. New code has unit tests; coverage ≥ 80% on new code (interval parser, runIngestion
   resilience/due-filter, cron auth). LLM/DB mocked; connectors stubbed.
