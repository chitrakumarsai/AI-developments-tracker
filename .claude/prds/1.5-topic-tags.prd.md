# PRD ‚ÄĒ Topic Tags (cross-source "what's this about?")

**Subphase:** 1.5 (filters/UX) ¬∑ **Type:** standard+ (new column + migration, deterministic tagging, new route, UI) ‚Üí `database-reviewer` on the migration + a light `security-reviewer` note (untrusted item text, matched read-only).

> ‚úÖ GATE 1 approved. **Tag source = keyword-only** (no LLM / no OpenAI cost); UX = chips + topic page + trending; scope = thin.

## Problem (from app-feedback-v1)
The feed is a per-category scroll. The user doesn't think in categories ‚ÄĒ they think in **events**: "Qwen3 dropped", "Molt Bot launched". They want to click that *thing* and see everything about it ‚ÄĒ the repo, the paper, the Reddit/HN discussion ‚ÄĒ gathered in one place, link-first. Today there's no cross-source connective tissue.

## Goal
Give each item **topic tags** (canonical entity/event names), and let the user click a topic to see **all items about it across every category**, newest first, with dates. Turn "an infinite sea" into "a finite set of things happening right now."

## Preferences (locked at GATE 1)
| Decision | Choice | Why |
|---|---|---|
| Tag source | **Keyword-only** ‚ÄĒ a curated map of model/lab/product patterns ‚Üí canonical slug. No LLM. | User: avoid OpenAI cost. Deterministic, free, testable. |
| Topic UX | **Topic pages** (`/topic/[slug]`) + **clickable topic chips** on cards + a small **Trending strip** | Matches the described click-through experience |
| Scope | **Thin**: tag ‚Üí chips ‚Üí topic page. Defer duplicate-collapsing + trend-rank tuning | Thin vertical slice (¬ß12.2) |

## Scope (this slice)
- New `items.topics text[]` column (canonical slugs), separate from coarse source `tags`.
- **Keyword tagging (deterministic):** match item title+summary against a curated map of known models/labs/products ‚Üí canonical slugs (e.g. `qwen`, `qwen3`, `llama`, `claude`, `deepseek`, `gemini`, `mistral`, `openai`, ‚Ä¶). No LLM, no API cost. Runs on ingest.
- Topic chips on `ItemCard` linking to `/topic/[slug]`.
- `/topic/[slug]` page: all items whose `topics` contains the slug, across categories, newest first, finite-first (reuse Show more).
- Minimal Trending strip on the home header: top N topics by (recent count), each linking to its page.

> **Coverage note:** keyword-only misses brand-new names until the map is updated. Accepted for this slice; a "suggested new topics" backlog (or opt-in LLM pass) can come later without rework ‚ÄĒ the `topics` column and UI are source-agnostic.

## Out of scope (follow-ups)
- Duplicate/near-duplicate collapsing of Reddit/HN posts (separate slice).
- Sophisticated trend ranking (velocity, decay), topic aliases/merge UI, per-user topic follows (Phase 2).
- Backfill tagging of the full historical corpus beyond a capped first pass.

## Data model
- `items.topics text[] default '{}'` (canonical slugs, e.g. `qwen3`, `molt-bot`). Indexed GIN for `@>` containment.
- Optional (deferred): a `topics` catalog table (slug‚Üílabel, first_seen, count). Slice 1 derives labels by prettifying slugs.

## Acceptance criteria
- [ ] Migration adds `items.topics` + GIN index; typed client updated.
- [ ] Keyword tagging populates topics for new items on ingest; slugs are canonical, deduped, `[a-z0-9-]`.
- [ ] Keyword map covers a seed set of known models/labs/products; adding a term is data, not new code paths.
- [ ] `ItemCard` renders up to N topic chips linking to `/topic/[slug]`.
- [ ] `/topic/[slug]` shows all matching items across categories, newest first, finite-first with Show more, with dates.
- [ ] Trending strip shows top topics and links through.
- [ ] Tests green (unit: keyword matcher + slug normalizer + topic query; component: chip render); tsc + lint clean.
- [ ] `database-reviewer` on the migration; light `security-reviewer` note (untrusted text, read-only match).
- [ ] Live-verified against cloud; a known model produces a topic page aggregating ‚Č•2 sources.

## Risks
- **Coverage gap** (novel names missed) ‚Üí curated map is easy to extend; accepted for thin slice; optional LLM pass later, no rework.
- **Topic fragmentation** (Qwen3 vs qwen-3) ‚Üí canonical slugging + one slug per concept in the map.
- **False positives** (short/ambiguous keywords) ‚Üí require word-boundary matches; keep the seed map specific.
