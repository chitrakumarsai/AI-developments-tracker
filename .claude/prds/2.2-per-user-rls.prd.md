# PRD — Phase 2.2: Per-user data + RLS

> Problem-first PRD for isolating personalization per account and enforcing it at
> the database. Pairs with `.claude/plans/2.2-per-user-rls.plan.md`. Depends on
> 2.1 (needs `auth.uid()`); branches off `feat/2.1-authentication`.

**Status:** Draft — awaiting GATE 1 approval
**Subphase:** 2.2 (Per-user data + RLS)
**Author:** Claude + owner preference-checkpoint (2026-07-05)
**Tier:** standard+ · `database-reviewer` **and** `security-reviewer` passes required

---

## 1. Problem

2.1 established identity but every signed-in user still shares one data plane:
feedback, saved views, and settings are global, and RLS is relaxed from Phase 1.
Before the site can serve more than the owner, each account's personalization must
be **private and enforced at the database** — not merely hidden in the UI. Without
row-level security, any user (or a leaked anon key) could read or write anyone
else's data. 2.2 makes personalization per-user and proves isolation.

## 2. Locked decisions (from the 2.2 preference-checkpoint, 2026-07-05)

| Decision | Choice | Consequence |
|---|---|---|
| **Existing Phase-1 data** | **Backfill to the owner account** | Pre-auth feedback/saved views + the singleton `app_settings` row are assigned to the owner's `user_id` in the migration, so the owner's tuning carries over. |
| **Feed read path** | **End-to-end RLS (anon-read policy)** | `sources`/`items` get an anon+auth read policy; feed reads move to the auth-aware SSR/anon client. Service-role drops out of the read path — it stays only for ingestion writes. (This is the anon-read policy deferred from 2.1 Step 6.) |
| **Owner identity in DB** | **`profiles` table with a `role` column** | `profiles(id → auth.users, role, email)`, upserted on sign-in; owner set from `OWNER_EMAIL`. RLS references `profiles.role = 'owner'` for source-catalog writes. Extensible to more roles later. |

## 3. Goals

- Add `user_id` to `feedback`, `saved_views`, `app_settings`, and read-state;
  convert `app_settings` from a singleton to one row per user.
- Introduce a `profiles` table populated on sign-in; owner derived from `OWNER_EMAIL`.
- Enable RLS with explicit policies on every user-scoped table so a user can only
  read/write their own rows.
- Anon + authenticated **read** on `sources`/`items`; **write** to the catalog is
  owner-only; `items` writes are service-role (ingestion) only.
- Move feed reads to the RLS-enforced client; keep ingestion on service-role.
- Scope ranking/feedback, saved views, and settings to the current user.

## 4. Non-goals (deferred)

- Production provisioning (prod project, SMTP, CAPTCHA, CSP) — 2.3.
- Roles beyond `owner`/`member`; team/sharing features.
- The in-app Source Management UI / automated discovery — 2.4.

## 5. Users & core value

Each account sees the **same shared, owner-curated feed** but its **likes, saved
views, settings, and read-state are private** and shape only *its* ranking. The
owner additionally curates the source catalog. Anonymous visitors keep read-only
access to the feed (2.1 access model), now backed by an explicit anon-read policy
rather than a relaxed table.

## 6. Success criteria

- **Cross-user isolation (the headline):** user A cannot read or write user B's
  feedback / saved views / settings / read-state — proven by tests, enforced by RLS.
- Owner-only catalog writes: a non-owner cannot insert/update/delete `sources`.
- Ingestion still writes `items` (service-role) and the cron path is unaffected.
- Anonymous `GET /` still renders the feed via the anon client (no service-role).
- The owner's Phase-1 feedback/views/settings survive the migration (backfill).
- `database-reviewer` + `security-reviewer` pass with no CRITICAL/HIGH open.

## 7. Key risks

1. **A missing or wrong policy leaks data** — every user-scoped table gets an
   explicit policy for each of select/insert/update/delete; the default-deny of RLS
   plus tests for each verb close gaps.
2. **Backfill misassigns rows** — backfill only touches `user_id IS NULL` rows and
   assigns the resolved owner id; runs once, idempotent, inside the migration.
3. **Read path regression** — moving feed reads to the anon client must not break
   anon browsing or the daily-cap/filter queries; covered by existing feed tests
   plus an anon-read check.
4. **`profiles` not populated** — if the upsert-on-sign-in fails, owner writes break.
   Upsert is idempotent and runs server-side after session establishment; owner
   role is re-derived from `OWNER_EMAIL` so it self-heals on next sign-in.
5. **Service-role bypass leaks** — service-role bypasses RLS by design; its use is
   confined to ingestion writes and audited so it never serves user reads.

## 8. Open questions (resolve during 2.2 planning)

- Whether `read-state` is its own table or a column set — confirm current schema in
