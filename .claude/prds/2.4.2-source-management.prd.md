# PRD — 2.4.2 In-app Source Management UI (Slice 1: Add + Ingest)

**Phase:** 2.4 (Discoverability & growth) · **Slice:** 2.4.2, Slice 1 of 2
**Status:** GATE 1 — awaiting plan approval
**Tier:** standard + **mandatory `security-reviewer`** (owner data · state-changing endpoints · untrusted ingestion)

---

## 1. Why

Adding the `eugeneyan` source this session required hand-writing SQL against the pooler
(`insert into sources …`) and hitting `/api/ingest/run` with the `CRON_SECRET` — friction the
product spec (§6 "Source Management UI", §14 subphase 2.4) says should live in the app. The owner
should curate the live `sources` catalog from the UI, with feed validation, without touching psql.

## 2. Preferences captured (preference-checkpoint, 2026-07-08)

| Decision | Choice |
|---|---|
| **First-slice scope** | **Add + ingest** — add source (feed-validated) → save → "Ingest now" → see status/last_fetched/item count. Pause/resume · re-weight · archive deferred to Slice 2. |
| **Location** | **Enhance the existing `/sources` page** — owner sees management affordances inline; members see the read-only catalog. |
| **Ingest UX** | **Await + toast result** — button awaits the run and shows `Pulled N items · M new`. |

## 3. Scope

### In scope (Slice 1)
- **Add a source** (owner-only): `name`, `url`, `category`, `ingestion_type`, `tags` → SSRF-guarded
  feed validation → insert an `active` `sources` row. Rejects unreachable/invalid feeds with a reason.
- **Live catalog list**: every source with `status`, `category`, `last_fetched`, **item count**, `priority`.
- **"Ingest now"** per source (owner-only): runs ingestion for that one source, awaits, shows
  `Pulled N items · M new` and refreshes `last_fetched` + count.
- **Owner-gating**: management controls render only for the owner; members/other signed-in users see
  the read-only catalog. Anon never reaches `/sources` (already `requireSession`-gated, `noindex`).

### Out of scope (→ Slice 2 / later)
- Pause/resume, re-weight priority, archive (soft-delete) + restore, edit metadata.
- Bulk actions, "ingest all", per-source refresh-interval editing.
- Automated citation/mention-trail discovery (separate 2.4 track).

## 4. The plumbing gap (drives the design)
`POST /api/ingest/run` is guarded by `isAuthorizedCron` (server-only `CRON_SECRET`) → **not callable
from the browser**. So the owner "Ingest now" action gets a **new owner-gated route** that calls the
same reusable `runIngestion(getServerClient(), { sourceId })` helper after `requireOwner`. Same
pattern as `/api/candidates/promote`. No secret ever reaches the client.

## 5. Reuse (no new deps)
- `validateFeedUrl()` (`lib/candidates/validate.ts`) — SSRF-guarded, RSS-parses, returns a reason.
- `promoteCandidate`'s insert shape (`lib/candidates/persist.ts`) → new `createSource()` helper.
- `runIngestion(client, { sourceId })` + `RunSummary` (`lib/ingestion/run.ts`).
- `requireOwner()` (`lib/auth/session.ts`), `isOwner`/role helpers (`lib/auth/role.ts`).
- `enforceRateLimit()` (`lib/rate-limit/limiter.ts`) — reuse the `"candidates"` owner bucket.
- `sanitizeText()` (`lib/ingestion/sanitize.ts`); `ApiResponse` envelope; existing `/sources` page +
  `AddCandidateForm` as the form pattern.
- Owner-only RLS writes on `sources` (2.2) remain the DB-level backstop; API authz = `requireOwner`.

## 6. Data
No schema change. Reads `sources` (`SourceRow`) + `items` counts grouped by `source_id`.

## 7. Acceptance criteria
1. Owner submits a valid RSS URL → new `active` source appears in the catalog; an invalid/unreachable
   URL is rejected with a human reason and **no row is written**.
2. Owner clicks "Ingest now" → items ingest → toast shows `Pulled N items · M new`; `last_fetched`
   and item count update on refresh.
3. A **non-owner** signed-in user sees the catalog **read-only** (no Add form, no Ingest buttons); a
   direct `POST` to either new route returns **403**. Anon → redirected to `/sign-in`.
4. Add/ingest endpoints are rate-limited and reject non-JSON / invalid payloads (400) and unsafe URLs
   (422) exactly like `/api/candidates/promote`.
5. All new code TDD'd, ≥80% coverage on new modules; `security-reviewer` passes with no CRIT/HIGH.

## 8. Risks
- **Service-role bypass**: new routes write via service-role `getServerClient()`, so `requireOwner` is
  the *only* authz gate — it must be airtight (security-reviewer focus). Established, but re-verified.
- **Ingest latency**: a slow feed holds the request open (await UX). Acceptable at current volume;
  bounded by the connector's existing fetch timeout. Revisit if catalog grows.
- **XSS**: source `name`/`category`/`tags` are owner-authored + `sanitizeText`'d on write and React-
  escaped on render. No `dangerouslySetInnerHTML`.
