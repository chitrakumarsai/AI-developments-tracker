## Problem

The feed covers papers + company blogs (all RSS). The next category — **GitHub Repositories** — has no RSS/feed; it needs an **API connector**. Per `api-connector-builder`, this must match the repo's existing pattern (the `Connector` contract + registry + sanitize + link-first + fixture tests), not introduce a new architecture. It's also the first `ingestion_type='api'` source, so the registry must learn to route among multiple API providers.

## Goal

Ingest **notable AI repositories** from the GitHub Search API as link-first feed items (category `GitHub Repositories`), dispatched through a new host-based `api` router — so adding Hugging Face next is just another host branch.

## Recorded preferences (preference-checkpoint, 2026-07-01)

| Decision | Choice |
|---|---|
| Content | Notable AI repos via GitHub Search API |
| API routing | Host-based router under `ingestion_type='api'` (no schema change) |
| Signal | Recently **active** (pushed ≤ ~30 days) + sorted by **stars** desc |

## Scope (this slice)

1. **`api` host router** in the registry: `api.github.com` → GitHub connector; unknown host → warning. (HF host added next slice.)
2. **GitHub connector** (`lib/ingestion/api/github.ts`) — `Connector`: query the Search API, map repos → `NormalizedItem[]`, sanitize, cap 50, link-first (`html_url`).
3. **Auth** via `GITHUB_TOKEN` (added to `.env.example`/`.env.local`); use it when present, warn-and-continue if absent (unauth search still works at lower limits).
4. **Seed** one GitHub source (`ingestion_type='api'`, category `GitHub Repositories`) whose `url` holds the topic query; connector injects the rolling `pushed:>=` date + `per_page`.
5. Capture the connector pattern as the project skill **`.claude/skills/ingestion-connector`** (specced in CLAUDE.md §16.2) so HF + future connectors reuse it.

## Non-goals (deferred)

- Hugging Face connector → next slice (reuses this router).
- Releases-of-watched-repos, GitHub topics beyond the seed query → later.
- Ranking/visibility of repos in the feed → 1.4/1.5 (same recency-sort caveat as blogs).

## Item mapping (link-first)

| NormalizedItem | GitHub Search field |
|---|---|
| `title` | `full_name` (e.g. `owner/repo`) |
| `url` | `html_url` |
| `summary` | `description` (sanitized) |
| `author` | `owner.login` |
| `publishedAt` | `pushed_at` |
| `tags` | `source.tags` |
| `category` | `source.category` (`GitHub Repositories`) |

## Success criteria

1. `ingestion_type='api'` sources route by host; `api.github.com` → GitHub connector.
2. A live run ingests real AI repos as items linking to github.com; re-run dedupes to 0 (unique `url`).
3. Repo `description` is sanitized (untrusted §12.7); no `dangerouslySetInnerHTML`.
4. Missing token → warning, not a crash; non-200 / rate-limit (403) → warning, not a throw (per-source resilience §12.7).
5. Connector + router + mapping unit-tested against a saved JSON fixture; existing tests still green; ≥80% coverage on new `lib/ingestion` code.
6. `GITHUB_TOKEN` only server-side (never `NEXT_PUBLIC_*`).

## Risks

- **Search API rate limits** (30/min authed, 10/min unauth) → one request/source/run; 403 → warning.
- **Query staleness** if a date were baked into the seed → connector computes the `pushed:>=` window at runtime; topics stay editable via `sources.url`.
- **`api-connector-builder` guardrail**: reuse the existing `Connector`/registry/sanitize pattern — the new connector must look native, not imported from another ecosystem.

## Acceptance demo

Add `GITHUB_TOKEN` → seed the GitHub source → `POST /api/ingest/run?source=<id>` → items appear linking to github.com repos → re-run adds 0.
